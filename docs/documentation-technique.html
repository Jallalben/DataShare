Section 01

Présentation générale

DataShare est une application web qui permet d'envoyer un fichier à quelqu'un en lui partageant un lien. Ce lien est temporaire : il expire automatiquement après une durée choisie par l'expéditeur, entre 1 et 7 jours. Une fois le délai dépassé, le fichier est supprimé du serveur — aucune donnée ne reste accessible ou stockée silencieusement.

L'application est destinée à des utilisateurs qui ont besoin d'un moyen rapide et sécurisé de partager des fichiers sans passer par email ou par un service cloud permanent.

Fonctionnalités principales

  • Inscription et connexion — chaque utilisateur dispose d'un compte personnel protégé par mot de passe.
  • Upload de fichiers — jusqu'à 50 Mo, avec choix de la durée d'expiration (1 à 7 jours).
  • Lien de partage — un lien unique est généré à chaque upload, accessible sans compte pour le destinataire.
  • Mon espace — tableau de bord personnel listant les fichiers actifs et expirés, avec possibilité de supprimer.
  • Purge automatique — les fichiers expirés sont supprimés automatiquement toutes les heures côté serveur.
Section 02

Architecture de l'application

L'application est découpée en trois blocs indépendants qui communiquent entre eux : l'interface utilisateur, le serveur, et la base de données. Chaque bloc tourne dans son propre conteneur Docker et peut être mis à jour sans toucher aux autres.

Flux d'une action typique

Quand un utilisateur uploade un fichier, voici ce qui se passe dans l'ordre :

  1. Le navigateur envoie le fichier au backend via une requête HTTP sécurisée.
  2. Le backend vérifie le token d'authentification, valide le fichier, puis le sauvegarde sur le disque.
  3. Un enregistrement est créé en base de données avec les métadonnées (nom, taille, date d'expiration) et un identifiant unique pour le lien de partage.
  4. Le lien est renvoyé au navigateur et affiché à l'utilisateur pour qu'il puisse le copier.
Le lien de partage fonctionne sans compte. Le destinataire clique sur le lien, voit les informations du fichier et peut le télécharger directement — sans inscription ni connexion.
Section 03

Justification des choix technologiques

NestJS pour le backend

J'ai choisi NestJS parce qu'il impose une structure claire dès le départ : chaque fonctionnalité (authentification, fichiers, tâches planifiées) vit dans son propre module indépendant. Cela rend le code facile à lire, à tester et à faire évoluer sans toucher au reste de l'application. NestJS est construit sur Node.js, ce qui permet de gérer plusieurs requêtes simultanées sans ralentissement, un point important pour une application de transfert de fichiers.

React pour le frontend

React est la bibliothèque la plus répandue pour construire des interfaces web interactives. L'interface de DataShare — modale d'upload, barre de progression, tableau de bord — est composée de blocs réutilisables. React permet de les assembler proprement et de mettre à jour uniquement ce qui change à l'écran, ce qui donne une navigation fluide sans rechargement de page.

PostgreSQL pour la base de données

PostgreSQL est une base de données relationnelle robuste et fiable, idéale pour stocker les informations structurées de DataShare : utilisateurs, fichiers, tokens de partage, dates d'expiration. Son système de contraintes garantit l'intégrité des données — par exemple, on ne peut pas créer un fichier sans l'associer à un utilisateur existant.

Docker pour l'infrastructure

Docker permet d'empaqueter l'application avec toutes ses dépendances dans des conteneurs isolés. Le résultat : l'application fonctionne de manière identique sur n'importe quelle machine, que ce soit pour le développement, les tests ou la production. Une seule commande suffit pour démarrer l'ensemble de la stack.

JWT pour l'authentification

Les tokens JWT (JSON Web Token) permettent de vérifier l'identité d'un utilisateur sans stocker d'information de session côté serveur. Le backend génère un token signé à la connexion. Ce token est ensuite présenté par le navigateur à chaque requête sensible — le serveur vérifie la signature et identifie l'utilisateur sans consulter la base de données.

GitHub Actions pour l'intégration continue

À chaque modification du code poussée sur le dépôt, GitHub Actions lance automatiquement l'ensemble des tests — unitaires, d'intégration et navigateur. Si un test échoue, le code ne peut pas être considéré comme stable. Ce processus automatique évite les régressions et garantit qu'une fonctionnalité qui marche continue de marcher après chaque évolution.

Section 04

Modèle de données

La base de données contient deux tables : les utilisateurs et les fichiers. Un utilisateur peut avoir plusieurs fichiers. Chaque fichier appartient à un seul utilisateur.

Table : users

Champ Type Rôle
id PK UUID Identifiant unique de l'utilisateur, généré automatiquement.
email Texte Adresse email, unique dans la base. Sert d'identifiant de connexion.
password Texte Mot de passe haché avec bcrypt. Jamais stocké en clair.
createdAt auto Date Date et heure de création du compte.

Table : files

Champ Type Rôle
id PK UUID Identifiant unique du fichier.
userId FK UUID Référence à l'utilisateur propriétaire du fichier.
originalName Texte Nom original du fichier tel qu'envoyé par l'utilisateur.
filename Texte Nom stocké sur le disque — UUID généré pour éviter les conflits.
mimetype Texte Type de contenu du fichier (ex. image/png, application/pdf).
size Nombre Taille du fichier en octets.
downloadToken UUID Identifiant unique du lien de partage. Inclus dans l'URL publique.
createdAt auto Date Date et heure de l'upload.
expiresAt Date Date à laquelle le lien expire et le fichier est supprimé.
La relation entre les deux tables : un utilisateur peut avoir plusieurs fichiers, mais chaque fichier appartient à un seul utilisateur. Si un utilisateur est supprimé, ses fichiers sont supprimés en cascade.
Section 05

Documentation des endpoints principaux

L'API dispose d'une documentation interactive générée automatiquement via Swagger. Elle est accessible à l'adresse http://localhost:3001/api/docs lorsque l'application tourne.

Cette interface liste tous les endpoints disponibles, détaille les paramètres attendus et les réponses possibles, et permet de tester chaque appel directement depuis le navigateur — sans outil externe. Pour les endpoints protégés, il suffit de cliquer sur Authorize en haut à droite et de coller le token JWT obtenu à la connexion.

Section 06

Sécurité et gestion des accès

Mots de passe

Les mots de passe ne sont jamais stockés tels quels. J'utilise bcrypt, un algorithme de hachage conçu spécifiquement pour les mots de passe. Même avec un accès direct à la base de données, un attaquant ne verrait qu'une suite de caractères illisible — retrouver le mot de passe original est impossible en pratique.

Sessions et tokens

Après connexion, l'application génère un token JWT signé, valable 24 heures. Ce token est présenté à chaque action sensible (upload, consultation de la liste, suppression). Le serveur vérifie la signature du token sans avoir à consulter la base de données à chaque fois. Passé 24h, le token expire et l'utilisateur doit se reconnecter.

Liens de partage

Chaque lien de partage contient un UUID v4 — un identifiant de 122 bits générés aléatoirement. La probabilité de tomber par hasard sur un lien valide est astronomiquement faible. Il faut connaître le lien exact pour accéder à un fichier.

Filtrage des fichiers uploadés

Les fichiers exécutables sont bloqués à l'upload — extensions .exe, .bat, .sh, .cmd, .msi, .ps1, .vbs — ainsi que les types MIME correspondants. La taille maximale acceptée est de 50 Mo par fichier.

Expiration automatique

Chaque fichier a une date d'expiration. Un processus automatique tourne toutes les heures et supprime définitivement les fichiers expirés — base de données et disque. Aucune donnée ne reste accessible après expiration.

Points d'amélioration identifiés pour une mise en production : limiter le nombre de tentatives de connexion par minute (rate limiting), restreindre les origines CORS aux domaines autorisés, et ajouter des en-têtes HTTP de sécurité via un middleware dédié.
Section 07

Qualité, tests et maintenance

Stratégie de tests

J'ai mis en place trois niveaux de tests qui se complètent :

  • Tests unitaires — vérifient la logique de chaque fonction isolément. 11 tests au total : 4 côté backend (Jest), 7 côté frontend (Vitest).
  • Tests d'intégration — vérifient que les endpoints API fonctionnent correctement avec une vraie base de données PostgreSQL.
  • Tests navigateur (E2E) — Cypress simule un utilisateur réel : inscription, connexion, upload, téléchargement. 6 scénarios couverts, exécutés dans un vrai navigateur.

Intégration continue

GitHub Actions lance automatiquement tous les tests à chaque modification du code. Si un test échoue, la modification est signalée immédiatement — avant qu'elle n'affecte qui que ce soit. Les tests E2E s'exécutent dans un environnement Docker identique à la production.

Maintenance

La purge automatique des fichiers expirés est la seule tâche de maintenance récurrente — elle est entièrement automatisée. Pour le reste, je vérifie les dépendances avec npm audit régulièrement pour détecter les failles de sécurité connues. Les mises à jour de l'application se déploient sans interruption grâce à Docker.

Versionnage

Chaque phase du projet correspond à un tag Git versionné : de v1.0-phase1-done à v1.0.0. Il est possible de revenir à n'importe quel état antérieur de l'application en quelques secondes.

Section 08

Installation et exécution

Prérequis

Une seule dépendance est nécessaire pour faire tourner l'application : Docker Desktop, disponible gratuitement sur Windows, Mac et Linux. Il n'y a rien d'autre à installer — ni Node.js, ni PostgreSQL, ni aucune autre dépendance.

Démarrage

Trois étapes depuis un terminal :

  1. Cloner le dépôt : git clone https://github.com/Jallalben/DataShare.git
  2. Copier la configuration : cp .env.example .env
  3. Démarrer : docker compose up --build

L'application est ensuite disponible sur http://localhost:3000. La première fois, Docker télécharge les images nécessaires — cela prend quelques minutes. Les démarrages suivants sont instantanés.

Commandes utiles

  • docker compose up — démarrer la stack
  • docker compose down — arrêter la stack
  • make e2e — lancer tous les tests E2E dans Docker
  • docker compose logs -f — suivre les logs en temps réel
Les données (utilisateurs, fichiers) sont conservées dans des volumes Docker persistants. Elles survivent aux redémarrages. Pour repartir d'une base propre : docker compose down -v.
Section 09

Utilisation de l'IA dans le développement

J'ai utilisé Claude (Anthropic) comme assistant tout au long du développement, via l'extension Claude Code dans Visual Studio Code. L'IA n'a pas écrit le projet à ma place — elle a joué le rôle d'un pair avec qui réfléchir, déboguer et avancer plus vite sur les parties techniques répétitives.

Ce que l'IA a fait

Génération de code boilerplate (configurations Docker, structure NestJS), aide au débogage des erreurs Cypress, rédaction des fichiers de documentation, suggestions sur les choix d'architecture.

Ce que j'ai fait

Toutes les décisions fonctionnelles, la validation de chaque bloc de code, les corrections quand l'IA se trompait, l'intégration et les tests manuels de chaque fonctionnalité.

Ce que l'IA ne fait pas bien

Elle ne connaît pas le contexte du projet sans qu'on le lui explique. Elle propose parfois des solutions trop complexes ou incorrectes — c'est au développeur de valider, corriger et adapter.

Ce que ça change concrètement

Les tâches répétitives (écrire des tests, configurer la CI, structurer la doc) prennent moins de temps. Cela laisse plus d'espace pour se concentrer sur la logique métier et la qualité du résultat final.

L'ensemble du code a été relu, compris et validé ligne par ligne. Aucun bloc de code n'a été intégré sans être testé manuellement dans l'application.
